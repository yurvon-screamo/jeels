# Дизайн-спецификация системы Jeers

## Введение

Система Jeers представляет собой минималистичный аналог Anki для интервального повторения материала. Она фокусируется на создании и управлении карточками, а также на процессе изучения и повторения. Архитектура построена в стиле Clean Architecture с использованием Domain-Driven Design (DDD).

## User Stories

Пользовательские истории сгруппированы по функциональным требованиям (FR) и описывают основные потребности пользователя.

### FR1: Создание и управление карточками

Это требование охватывает добавление, редактирование и удаление контента для изучения в едином пространстве.

- Как пользователь, я хочу создавать новые карточки с вопросом и ответом, чтобы добавлять материал для запоминания.
- Как пользователь, я хочу редактировать существующие карточки, чтобы исправлять ошибки или обновлять информацию.
- Как пользователь, я хочу удалять карточки, чтобы избавляться от ненужного контента.

### FR2: Изучение и повторение карточек

Это требование фокусируется на взаимодействии с карточками и автоматизации повторений.

- Как пользователь, я хочу просматривать карточки в режиме изучения, чтобы видеть вопрос, а затем ответ.
- Как пользователь, я хочу оценивать свой ответ (например, "легко", "хорошо", "трудно", "забыл"), чтобы приложение учитывало мою уверенность в запоминании.
- Как пользователь, я хочу, чтобы приложение автоматически планировало повторения на основе моих оценок, чтобы я повторял материал в оптимальное время и не забывал его.

## Сценарии использования (Use Cases)

Каждый сценарий описывает актера, предусловия, основной поток, альтернативные потоки и постусловия. Они полностью покрывают пользовательские истории.

### FR1: Создание и управление карточками

#### Use Case 1: Создание новой карточки

- **Актер**: Пользователь.
- **Предусловия**: Пользователь находится в разделе управления карточками.
- **Основной поток**:
  1. Пользователь выбирает опцию "Создать карточку".
  2. Пользователь вводит вопрос и ответ.
  3. Пользователь подтверждает создание.
- **Альтернативные потоки**:
  - Если поля пустые, система показывает ошибку и возвращает к шагу 2.
  - Если карточка с таким же вопросом уже существует (проверка регистронезависимая, с учетом обрезки пробелов), система показывает ошибку дубликата и возвращает к шагу 2.
- **Постусловия**: Новая карточка добавлена в пространство, отображается подтверждение.

#### Use Case 3: Редактирование существующей карточки

- **Актер**: Пользователь.
- **Предусловия**: Карточка существует, пользователь в списке карточек.
- **Основной поток**:
  1. Пользователь выбирает карточку для редактирования.
  2. Пользователь изменяет вопрос, или ответ.
  3. Пользователь подтверждает изменения.
- **Альтернативные потоки**:
  - Если изменений нет, система предлагает отменить.
  - Если новый вопрос дублирует вопрос другой существующей карточки (проверка регистронезависимая, с учетом обрезки пробелов), система показывает ошибку дубликата и возвращает к шагу 2.
- **Постусловия**: Карточка обновлена, отображается актуальная версия.

#### Use Case 4: Удаление карточки

- **Актер**: Пользователь.
- **Предусловия**: Карточка существует, пользователь в списке карточек.
- **Основной поток**:
  1. Пользователь выбирает карточку для удаления.
  2. Пользователь подтверждает действие (с предупреждением).
- **Альтернативные потоки**: Если пользователь отменяет, возвращение к списку.
- **Постусловия**: Карточка удалена, список обновлен.

### FR2: Изучение и повторение карточек

#### Use Case 5: Просмотр карточки в режиме изучения

- **Актер**: Пользователь.
- **Предусловия**: Есть карточки, пользователь начинает сессию изучения.
- **Основной поток**:
  1. Система показывает вопрос.
  2. Пользователь думает над ответом.
  3. Пользователь запрашивает показ ответа.
  4. Система отображает ответ.
- **Постусловия**: Пользователь переходит к оценке ответа.

#### Use Case 6: Оценка ответа на карточку

- **Актер**: Пользователь.
- **Предусловия**: Пользователь просмотрел карточку и увидел ответ.
- **Основной поток**:
  1. Система предлагает варианты оценки ("легко", "хорошо", "трудно", "забыл").
  2. Пользователь выбирает оценку.
  3. Система сохраняет выбор.
- **Альтернативные потоки**: Если оценка не выбрана, система напоминает и ждет.
- **Постусловия**: Оценка зафиксирована, готовится следующая карточка.

#### Use Case 7: Начало сессии изучения

- **Актер**: Пользователь.
- **Предусловия**: Пользователь авторизован, есть карточки в системе.
- **Основной поток**:
  1. Пользователь запрашивает начало сессии изучения.
  2. Система находит все карточки, готовые к повторению (next_review_date <= текущее время).
  3. Система возвращает список карточек для изучения.
- **Альтернативные потоки**: Если нет карточек, готовых к повторению, система возвращает пустой список.
- **Постусловия**: Пользователь получает список карточек для изучения.

#### Use Case 8: Автоматическое планирование повторений

- **Актер**: Пользователь.
- **Предусловия**: Пользователь оценил карточку (Use Case 6).
- **Основной поток**:
  1. Система получает оценку карточки и предыдущее состояние памяти.
  2. Система рассчитывает интервал повторения и новое состояние памяти через SRS Service (FSRS).
  3. Система создает Review с оценкой и интервалом.
  4. Система обновляет карточку: добавляет Review, устанавливает next_review_date, stability и memory_state.
  5. Система сохраняет изменения.
- **Альтернативные потоки**: Если расчет SRS не удался, система возвращает ошибку.
- **Постусловия**: Карточка запланирована на следующее повторение, состояние памяти обновлено.

## Доменная модель

Модель построена по принципам DDD. Агрегатный корень — User. Все операции обеспечивают согласованность.

### Сущности

1. **User (Агрегатный корень, Entity)**  
   - **Описание**: Владелец карточек и процессов обучения.  
   - **Атрибуты**:  
     - id: Ulid.  
     - username: String.  
     - cards: HashMap<Ulid, Card>.  
   - **Методы**:  
     - new(username: String): User.  
     - id(): Ulid.  
     - username(): &str.  
     - cards(): &HashMap<Ulid, Card>.  
     - create_card(question: Question, answer: Answer): Result<Card, JeersError> (проверяет дубликаты по вопросу, регистронезависимо, с учетом обрезки пробелов).  
     - edit_card(card_id: Ulid, new_question: Question, new_answer: Answer): Result<(), JeersError> (проверяет дубликаты по новому вопросу, исключая текущую карточку).  
     - delete_card(card_id: Ulid): Result<(), JeersError>.  
     - start_study_session(): Vec<Card> (возвращает карточки, готовые к повторению).  
     - rate_card(card_id: Ulid, rating: Rating, interval: Duration): Result<(), JeersError>.  
     - schedule_next_review(card_id: Ulid, next_review_date: DateTime<Utc>, stability: Stability, memory_state: MemoryState): Result<(), JeersError>.  
     - get_card(card_id: Ulid): Option<&Card>.

2. **Card (Дочерняя сущность, Entity)**  
   - **Описание**: Флеш-карточка.  
   - **Атрибуты**:  
     - id: Ulid.  
     - question: Question.  
     - answer: Answer.  
     - reviews: VecDeque<Review>.  
     - next_review_date: DateTime<Utc>.  
     - stability: Stability.  
     - memory_state: Option<MemoryState> (состояние памяти для алгоритма FSRS).  
   - **Методы**:  
     - new(question: Question, answer: Answer): Card.  
     - id(): Ulid.  
     - question(): &Question.  
     - answer(): &Answer.  
     - reviews(): &VecDeque<Review>.  
     - next_review_date(): DateTime<Utc>.  
     - stability(): Stability.  
     - memory_state(): Option<MemoryState>.  
     - edit(new_question: Question, new_answer: Answer) (внутренний метод).  
     - add_review(review: Review) (внутренний метод).  
     - update_schedule(next_review_date: DateTime<Utc>, stability: Stability) (внутренний метод).  
     - update_memory_state(memory_state: MemoryState) (внутренний метод).  
     - is_due(): bool (проверяет, готова ли карточка к повторению).  
     - last_review_date(): Option<DateTime<Utc>> (возвращает дату последнего повторения).

3. **Review (Дочерняя сущность, Entity)**  
   - **Описание**: Запись оценки.  
   - **Атрибуты**:  
     - id: Ulid.  
     - rating: Rating.  
     - timestamp: DateTime<Utc>.  
     - interval: Duration.  
   - **Методы**:  
     - new(rating: Rating, interval: Duration): Review.  
     - id(): Ulid.  
     - rating(): Rating.  
     - timestamp(): DateTime<Utc>.  
     - interval(): Duration.

### Value Objects (Неизменяемые объекты)

- **Question**:  
  - text: String.  
  - Валидация: текст не может быть пустым (после trim).  
  - Методы: new(text: String): Result<Question, JeersError>, text(): &str.

- **Answer**:  
  - text: String.  
  - Валидация: текст не может быть пустым (после trim).  
  - Методы: new(text: String): Result<Answer, JeersError>, text(): &str.

- **Rating**:  
  - Enum с вариантами: Easy, Good, Hard, Again.

- **Duration**:  
  - days: u32.  
  - Методы: new(days: u32): Duration, days(): u32.

- **Stability**:  
  - value: f64.  
  - Валидация: значение не может быть отрицательным.  
  - Методы: new(value: f64): Result<Stability, JeersError>, value(): f64.

- **MemoryState**:  
  - stability: Stability.  
  - difficulty: f64.  
  - Валидация: difficulty не может быть отрицательным.  
  - Методы: new(stability: Stability, difficulty: f64): Result<MemoryState, JeersError>, stability(): Stability, difficulty(): f64.  
  - Описание: Состояние памяти для алгоритма FSRS, хранит стабильность и сложность карточки.

- **Ulid**:  
  - Используется для уникальных идентификаторов (вместо UUID).

- **DateTime<Utc>**:  
  - Используется из библиотеки chrono для работы с датами и временем.

## Обработка ошибок

Система использует единый тип ошибок `JeersError` для всех слоев приложения. Это обеспечивает согласованную обработку ошибок и упрощает их передачу между слоями.

### Типы ошибок

Все ошибки представлены в виде enum `JeersError` со следующими вариантами:

1. **UserNotFound { user_id: Ulid }**  
   - **Описание**: Пользователь с указанным идентификатором не найден.  
   - **Контекст**: Возникает при попытке выполнить операцию с несуществующим пользователем.  
   - **Сообщение**: "User with id {user_id} not found"

2. **UserNotFoundByUsername { username: String }**  
   - **Описание**: Пользователь с указанным именем не найден.  
   - **Контекст**: Возникает при поиске пользователя по имени.  
   - **Сообщение**: "User with username {username} not found"

3. **CardNotFound { card_id: Ulid }**  
   - **Описание**: Карточка с указанным идентификатором не найдена.  
   - **Контекст**: Возникает при попытке выполнить операцию с несуществующей карточкой (редактирование, удаление, оценка, планирование повторения).  
   - **Сообщение**: "Card with id {card_id} not found"

4. **DuplicateCard { question: String }**  
   - **Описание**: Карточка с указанным вопросом уже существует.  
   - **Контекст**: Возникает при создании новой карточки или редактировании существующей, если вопрос дублирует вопрос другой карточки. Проверка выполняется регистронезависимо с учетом обрезки пробелов. При редактировании текущая карточка исключается из проверки.  
   - **Сообщение**: "Card with question '{question}' already exists"

5. **InvalidQuestion { reason: String }**  
   - **Описание**: Некорректный вопрос.  
   - **Контекст**: Возникает при создании или редактировании карточки, если текст вопроса пустой (после trim).  
   - **Сообщение**: "Invalid question: {reason}"

6. **InvalidAnswer { reason: String }**  
   - **Описание**: Некорректный ответ.  
   - **Контекст**: Возникает при создании или редактировании карточки, если текст ответа пустой (после trim).  
   - **Сообщение**: "Invalid answer: {reason}"

7. **InvalidStability { reason: String }**  
   - **Описание**: Некорректное значение стабильности.  
   - **Контекст**: Возникает при попытке установить отрицательное значение стабильности.  
   - **Сообщение**: "Invalid stability: {reason}"

8. **InvalidDifficulty { reason: String }**  
   - **Описание**: Некорректное значение сложности.  
   - **Контекст**: Возникает при попытке установить отрицательное значение сложности в состоянии памяти.  
   - **Сообщение**: "Invalid difficulty: {reason}"

9. **InvalidMemoryState { reason: String }**  
   - **Описание**: Некорректное состояние памяти.  
   - **Контекст**: Возникает при создании или преобразовании состояния памяти с невалидными параметрами.  
   - **Сообщение**: "Invalid memory state: {reason}"

10. **SrsCalculationFailed { reason: String }**  
    - **Описание**: Ошибка расчета алгоритма интервального повторения (FSRS).  
    - **Контекст**: Возникает при сбое в работе библиотеки fsrs-rs (инициализация, расчет следующего состояния, преобразование данных).  
    - **Сообщение**: "SRS calculation failed: {reason}"

11. **RepositoryError { reason: String }**  
    - **Описание**: Ошибка работы с хранилищем данных.  
    - **Контекст**: Возникает при операциях с базой данных (поиск, сохранение, удаление).  
    - **Сообщение**: "Repository error: {reason}"

### Использование ошибок

- Все методы доменных сущностей и use cases возвращают `Result<T, JeersError>`.  
- Ошибки сериализуются и десериализуются для передачи между слоями.  
- Ошибки реализуют трейты `Display`, `Error` для удобной обработки и логирования.  
- Валидация на уровне домена возвращает соответствующие ошибки вместо паники.

## Компоненты системы

Архитектура основана на Clean Architecture с тремя слоями. Описана в нотации LikeC4 (DSL для визуализации). Интеграция с FSRS (Rust, fsrs crate версии 5.2) и SurrealDB (Rust SDK версии 2.3, локальное хранилище на RocksDB).
